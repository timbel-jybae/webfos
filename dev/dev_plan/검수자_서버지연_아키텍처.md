# 검수자 서버 측 지연 아키텍처

## 1. 개요

- **목적**: 클라이언트 버퍼링의 한계(화질 저하, 오디오 지연 실패)를 해결
- **방식**: Agent(서버)에서 HLS를 버퍼링하고, 지연된 스트림을 별도 Ingress로 제공
- **효과**: 검수자는 원본 화질 + 비디오/오디오 동기 지연

## 2. 아키텍처

```
[ HLS Source ] ----+----> [ LiveKit Ingress 1 ] ----> Room (실시간)
                   |           participant: ingress-hls-source
                   |
                   +----> [ HLS Delay Proxy ] ----> [ LiveKit Ingress 2 ] ----> Room (3.5초 지연)
                              (서버 버퍼)              participant: ingress-hls-delayed

Room 참가자:
- 참가자 1~3: ingress-hls-source 구독 (실시간)
- 검수자: ingress-hls-delayed 구독 (지연)
```

## 4. HLS Delay Proxy 상세

- **입력**: `HLS_URL` (환경변수, 실시간 소스와 동일)
- **출력**: `http://localhost:9999/playlist.m3u8` (지연된 플레이리스트)
- **로직**: 세그먼트 fetch → 메모리 버퍼 → 3.5초 후 서빙
- **의존성**: `m3u8`, `aiohttp`, `uvicorn`
